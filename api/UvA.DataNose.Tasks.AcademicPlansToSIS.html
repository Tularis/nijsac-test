Namespace UvA.DataNose.Tasks.AcademicPlansToSIS

Classes

ActivePeriods

Holds the active periods of a single student for a single programme-plan combination. Answers these questions:

  • Is date X within an active period of student Y for plancode Z? (yes / no)
  • Suggest replacement date for X where student Y is active for plancode Z (date)

The list of active periods for a single student and plancode is cached, so lookups will be efficient if consecutive questions are about the same student and plancode.

Assumption: active periods do not overlap! They can border on eachother. But in case of overlap, results will not be correct.

ExcludeCoursesInSIS

Safety measure: Exclude courses in SIS that according to SPA should not be there. Unlike most classes in this folder: Input is not a SPA-query, but a SPA-SIS Plan Compare.

ILOMinorsWarning

This class knows - by hardcoded configuration - a short list of ILO minors that need to be mentioned on the diploma itself, because it results in a teaching license at VWO level. It occurs a few times a year that an FNWI student completes such a minor. The servicedesk needs to be alerted when this happens because they have to (i) ask the ILO to attach the subplan (ii) alert bureau examens.

At the moment this class just logs warnings to a TaskRunner. ...(Further possibilities: send a mail somewhere? Add a special "ILO Minor" checklist item to the Graduation Checklists?) Separate warnings are logged for:

  • ILO minor is in current students plan
  • Current Student has enrolment for ILO minor but it is not in this plan (reasons can be good or bad)

This class works in three phases:

  1. Start work on a new student-plan combination
  2. Repeated: Check a minor found in the plan, log a warning if it's an ILO minor
  3. Finish work on this student-plan: log warnings for any remaining ILO minors for this student (he enrolled but didn't plan)

Steps 1-3 are typically repeated for each student-plan to check. The class is initialized by loading all known ILO minor enrolments for FNWI students in memory for quick lookup. It is therefore best used in a batch process where many plans are checked.

A summary of the issued ILO-minor warnings (including a count) will be kept, you can ask to add it to the log (e.g. at the end).

LocationMemory

Various PlanChangesToSIS subclasses use this class to remember which locations in the ADV already have an Advisement Override of type CourseDirective for the current student and programme/plan/subplan.

Reason: there can be max 1 advisement override for a given student, programme/plan/subplan and location. There are 3 types of plan changes that may need to add their (excluded / allowed) courses to this 1 advisent override.

We follow this policy:

  • The first plan-change that writes to SIS for the given student and location does so in "update" mode: if an advisement override already exitst for that location its contents are overwritten.
  • Any following plan changes that need to write to SIS for the same student and location do so in "merge" mode: the new allowed / excluded courses are added to the ones who are already there.

This class helps by just remembering whether a location was already treated. A fresh object of this class should be used for each student + plan in the main loop of a "ToSIS" batch.

SpaSisPlanCompare

Compare SPA and SIS versions of a students plan, e.g. to check results of the SPA-to-SIS transfers.

SpaSisPlanCompare.CompareResult

Result of the SpaSisPlanCompare operation: OK or not. We may add specific properties for various kinds of errors to enable automatic fixing of such errors. At the moment there is just one such property.
